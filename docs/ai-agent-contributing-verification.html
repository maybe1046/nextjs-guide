// agent-assisted code review

Verifying Human Changes
against CONTRIBUTING.md

Approaches for inserting an AI code agent as a gatekeeper that verifies pull requests and commits authored by human developers against the rules in CONTRIBUTING.md. The agent never authors the change — it only reviews it. The organising axis is where in the workflow the check runs; each point is a place to install a verification gate.

00How the agent gets its inputs

Every approach below feeds the agent the same three things:

InputSource
The changegit diff --staged, git diff main...HEAD, or gh pr diff <n> / the PR API.
The rulesCONTRIBUTING.md, plus docs/best-practices.html for the why behind borderline calls.
The outputStructured findings keyed to section numbers (§1–§9), each with a severity and an overall pass/fail verdict.

01Approaches by workflow stage

Ordered earliest to latest. A advisory stage informs but cannot stop a change; a blocking stage can reject it.

stage 1In-editor — as the human types advisory

  • IDE extension review (VS Code / JetBrains Claude Code) — the agent reviews the working diff in the editor before any commit. Earliest possible feedback, but purely advisory.

stage 2Pre-commit — local git hook invoking a headless agent blocking

  • pre-commit hook → claude -p (wired via husky or lefthook): the hook runs the agent over git diff --staged and exits non-zero on violations, blocking the commit locally.
  • commit-msg hook → agent: verify the message against Conventional Commits (§2) — though commitlint does this deterministically and far more cheaply.

stage 3Pre-push — local git hook invoking a headless agent blocking

  • pre-push hook → claude -p: reviews the whole branch diff (git diff main...HEAD) before it leaves the machine. Catches what per-commit review misses — e.g. §1 cross-package import boundaries that are only visible across multiple files.

stage 4On-demand — inside the developer's own Claude Code session advisory

  • Custom slash command — /check-contributing the developer runs before pushing; diffs their branch against the rulebook.
  • Custom subagent — a contributing-reviewer agent invoked with clean context for an unbiased audit.
  • Skill — a contributing-enforcer skill that auto-triggers when relevant files are staged.
  • Built-in /review — pointed at CONTRIBUTING.md.

stage 5PR-time CI — agent reviews the pull request blocking

  • GitHub Actions + Claude Code Action on pull_request — the agent reviews the PR diff, posts inline review comments citing §-numbers, and sets a check status.
  • GitLab CI / Bitbucket Pipelines / Jenkins — the same pattern, running claude -p in headless mode inside the pipeline.
  • Required status check + branch protection — make that agent job a required check so the merge is blocked until it passes. This is the only setup that truly forces the rules on human contributors.

stage 6PR via mention / bot advisory

  • @claude mention — the author or a human reviewer pings the Claude GitHub App to request a review against the rulebook.
  • Auto-review bot — a bot posts a review automatically on every PR open / update.

stage 7Server-side — before the push is even accepted blocking

  • pre-receive hook on a self-hosted git server (GitHub Enterprise / GitLab / Gitea) → headless agent. Rejects the push server-side; humans cannot bypass it by skipping local hooks with --no-verify.

stage 8Post-merge / scheduled advisory

  • Cron / scheduled agent — periodically audits main or recently merged PRs against CONTRIBUTING.md. Catches drift and code merged before enforcement existed; opens issues for violations.

02Cross-cutting building blocks

  • Headless wrapper (Claude Agent SDK / claude -p) — wrap the agent so it takes a diff plus the rulebook and emits structured pass/fail JSON. This one component plugs into stages 2, 3, 5, 7 and 8.
  • MCP server — exposes a validate-against-contributing tool that any MCP-capable agent can call.

03Advisory vs. blocking — what actually forces the rules

Stages 1, 4, 6 and 8 are advisory — they inform but do not stop anything. Stages 2, 3, 5 and 7 can block. But only a required CI check (stage 5) or a pre-receive hook (stage 7) is genuinely unbypassable by a human — local git hooks can always be skipped with git commit --no-verify.

Pick your gate deliberately If the goal is enforcement and not just a hint, the verification must live somewhere the developer cannot opt out of. That means the server: a required PR check or a pre-receive hook.

04Agent vs. deterministic — don't pay for what a linter does free

Route the lintable subset of CONTRIBUTING.md to deterministic tools, and give the agent only the judgment calls. Run the deterministic gate first, then let the agent review what remains — cheaper, and its findings won't drown in lint noise.

Deterministic (ESLint / tsc / commitlint) Needs an AI agent (judgment)
§5 no any, import type §1 import-boundary intent; cross-feature reach-ins
§2 Conventional Commit format §3 "useEffect only for synchronisation"
§8 naming patterns §4 ""use client" pushed as far down as possible"
§5 strict-mode flags §7 "a bug fix has a test that fails without the fix"

TL;DRRecommendation for this repo

The highest-leverage setup, in order:

1 · headless wrapper 2 · required PR check 3 · deterministic gate first
  1. Build the headless wrapper once — it is reused by every blocking stage.
  2. Wire stage 5 as a required status check — this forces the rules on every human PR and is the only unbypassable gate that fits a GitHub-hosted repo.
  3. Run ESLint / tsc / commitlint before the agent so it only judges the rules a linter cannot.
