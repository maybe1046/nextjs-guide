React · Next.js · Turborepo
Monorepo Best Practices

A consolidated, opinionated style guide for building and maintaining a React / Next.js application inside a Turborepo monorepo. The companion CONTRIBUTING.md is the enforced version of this document; the lint and TypeScript configs are the machine-enforced version.

00Authoritative sources

There is no longer a single official "React style guide" — react.dev replaced it. Treat the lint config as the enforced style guide and this document as the human-readable rationale.

01Monorepo & Turborepo structure

Layout

apps/            # deployable things — Next.js apps, thin composition only
  web/
  admin/
packages/        # everything reusable
  ui/            # shared React component library / design system
  eslint-config/      # shared flat ESLint config
  typescript-config/  # shared tsconfig bases
  utils/         # framework-agnostic helpers
  types/         # shared TS types / Zod schemas

Hard rules

  • Apps never import from apps. Apps import from packages/*. Packages never import from apps. Enforce this with eslint-plugin-boundaries, not just docs.
  • Use the "internal packages" / Just-in-Time pattern. Packages export raw .ts/.tsx and are compiled by the consuming app (Next.js transpilePackages) — no per-package build step.
  • Explicit exports maps, no mega-barrels. Barrel index.ts files that re-export everything wreck tree-shaking and slow builds. Export each module path.
  • One version of React across the repo. Pin it; use pnpm overrides + syncpack to keep versions aligned.
  • Dependencies live where they are used. Each package declares its own deps; don't dump everything in the root package.json.
  • Share configs as packages (@repo/typescript-config, @repo/eslint-config). No copy-pasted config files.
  • Use pnpm workspaces — best monorepo support, strict node_modules.
turbo.json Declare inputs/outputs precisely so caching is correct. Mark dev as "cache": false, "persistent": true. Put every consumed env var in globalEnv/env so the cache never serves a stale build.

02React

  • Function components + hooks only. No class components.
  • Don't use useEffect for things that aren't synchronization with an external system. Derived state → compute during render. Event responses → event handlers.
  • Keep components small and single-purpose. Extract when a component does two things.
  • Composition over prop-drilling. children and slots before context; context before a global store.
  • Lift state only as high as needed, no higher. Co-locate state with its consumer.
  • Stable keys — never the array index for dynamic lists.
  • Don't over-memoize. useMemo/useCallback/memo solve measured problems. With the React Compiler, prefer letting it handle memoization.
  • Components must be pure during render — no side effects, no mutation of props/state.
  • Custom hooks for reusable stateful logic; prefix with use.

✓ Derive during render

const fullName = `${first} ${last}`;
const visible = items.filter(i => i.active);

✗ Sync derived state via effect

useEffect(() => {
  setFullName(`${first} ${last}`);
}, [first, last]);

03Next.js — App Router

  • Server Components by default. Add "use client" only at the leaf that genuinely needs interactivity or browser APIs — push it as far down the tree as possible.
  • Fetch data in Server Components, close to where it's used. Don't lift all fetching to the page.
  • Know the caching layers — Request Memoization, Data Cache, Full Route Cache, Router Cache. Be explicit with fetch options and revalidateTag/revalidatePath. Defaults changed across Next 14 → 15; know your version.
  • Server Actions for mutations. Validate every input with Zod inside the action; treat all inputs as untrusted; never expose secrets.
  • Use the file conventions: loading.tsx, error.tsx, not-found.tsx, route.ts, layout.tsx.
  • next/image, next/font, next/link — always. No raw <img> for content images.
  • Stream slow data with <Suspense> instead of blocking the route.
  • Keep middleware.ts thin — it runs on every request.
  • Only NEXT_PUBLIC_* env vars reach the client. Validate all env at startup with Zod.

04TypeScript

  • strict: true plus noUncheckedIndexedAccess and noImplicitOverride, in a shared base tsconfig.
  • No any. Use unknown + narrowing. Ban any via lint.
  • Validate all external data (API responses, env, form input, searchParams) with Zod; derive types via z.infer.
  • Type props explicitly; avoid React.FC — prefer a plain function with typed props.
  • Discriminated unions for component variants and state machines.
  • Co-locate types with their feature; only truly shared types go in packages/types.
  • Use import type { ... } for type-only imports (lint rule consistent-type-imports).

05Tooling & enforcement

ESLint 9 flat config typescript-eslint eslint-plugin-react-hooks jsx-a11y Prettier / Biome Husky + lint-staged commitlint knip syncpack Changesets
  • ESLint flat config + typescript-eslint + eslint-plugin-react-hooks + jsx-a11y + @next/eslint-plugin-next. The hooks rules are non-negotiable.
  • Prettier for formatting — or Biome if you want lint + format in one fast tool.
  • Enforce monorepo boundaries with eslint-plugin-import / eslint-plugin-boundaries.
  • lint-staged + Husky (or lefthook) pre-commit; commitlint + Conventional Commits.
  • knip finds unused files/exports/deps. syncpack keeps versions consistent.
  • CI runs turbo lint check-types test build with caching + --filter so PRs only rebuild affected packages.

06State, data fetching & styling

  • Server state (anything from an API): fetch in Server Components, or TanStack Query for client-heavy apps. Never store server data in Redux/Zustand.
  • Client/UI state: useState/useReducer → context → Zustand for genuinely global state. Avoid Redux without a real reason.
  • URL as state — filters, tabs, pagination belong in searchParams.
  • Styling: Tailwind CSS (share the config as a package) or CSS Modules. Avoid runtime CSS-in-JS in the App Router — prefer zero-runtime (Tailwind, CSS Modules, vanilla-extract, Panda).
  • For a shared component library use CVA for variants; consider shadcn/ui for packages/ui.

07Testing

  • Vitest (or Jest) + React Testing Library — test behavior, not implementation.
  • Playwright for E2E, per app.
  • MSW to mock network at the boundary.
  • Pure functions get unit tests; Server Components/Actions get integration-style tests.
  • Wire tests into Turborepo so only affected packages run in CI.
  • Optional: Storybook for packages/ui + visual regression.

08Performance

  • Watch the client/server boundary — every "use client" ships JS. Audit with @next/bundle-analyzer.
  • next/dynamic for heavy client-only components; lazy-load below the fold.
  • Optimize images (next/image with proper sizes); self-host fonts via next/font.
  • Stream with <Suspense> so slow data doesn't block first paint.
  • Monitor Core Web Vitals (Speed Insights / useReportWebVitals).
  • Keep middleware minimal — it's on the hot path.

  • One feature = one folder; co-locate component + test + styles + sub-components.
  • Absolute imports via tsconfig paths (@/components) and workspace aliases — no ../../../.
  • Export explicitly; avoid deep barrel files.

10Git & CI

  • Conventional Commits + Changesets for versioning/changelogs of shared packages.
  • PRs gated on turbo lint check-types test build with remote caching enabled.
  • Trunk-based or short-lived branches; CODEOWNERS per apps/ and packages/.
  • A root CONTRIBUTING.md documents the structure and the "apps don't import apps" rule — that file is the enforced version of this guide.

TL;DR — adoption order

  1. strict TypeScript + shared tsconfig/eslint config packages.
  2. eslint-plugin-react-hooks + jsx-a11y + import boundaries — enforce, don't just document.
  3. Server Components by default; push "use client" to leaves.
  4. Zod-validate every external boundary.
  5. Correct turbo.json inputs/outputs/env for trustworthy caching.
  6. knip + syncpack so the monorepo doesn't rot.
