1. Mental model

The App Router flips the default: everything is a Server Component unless you opt in to client. Most of your tree should render on the server, fetch its own data, and ship zero JS. Client Components are islands of interactivity embedded in that server-rendered tree.

The three rules of React (memorize these)

  1. Components and hooks must be pure. No mutation of props, state, or external values during render. Same inputs ⇒ same output.
  2. React calls components and hooks. Never call a component as a function (MyComp()) and never call hooks conditionally or inside loops.
  3. Rules of hooks. Hooks only run at the top of a Component or another Hook. Use the ESLint plugin — don't argue with it.

Server vs Client — when to switch

Use a Server Component when…Use a Client Component when…
Fetching data, reading cookies, accessing the DB, rendering Markdown, anything that touches secrets. You need useState, useEffect, event handlers, browser APIs, or third-party hooks.
The component is mostly static or driven by URL state. The component is the leaf of an interactive widget (button, dropdown, form input).
Rule of thumb Push "use client" as far down the tree as possible. A <SubmitButton> can be a Client Component while the surrounding form, page, and layout stay on the server.

2. File & folder conventions

Colocate by feature inside app/. Shared building blocks live at the root. Use the @/ alias for absolute imports — never relative paths that climb more than one level (../../../ is a smell). 

apps/web/
├─ app/
│  ├─ (marketing)/              # route group — does not affect URL
│  │  └─ page.tsx
│  ├─ dashboard/
│  │  ├─ layout.tsx             # nested layout, preserves state on nav
│  │  ├─ page.tsx               # Server Component
│  │  ├─ loading.tsx            # Suspense fallback
│  │  ├─ error.tsx              # Error boundary (must be "use client")
│  │  └─ _components/           # feature-local, leading _ excludes from routing
│  │     └─ revenue-chart.tsx
│  ├─ api/
│  │  └─ webhooks/route.ts
│  ├─ layout.tsx                # root layout
│  └─ globals.css
├─ components/
│  └─ ui/                       # shadcn-style design primitives
│     ├─ button.tsx
│     └─ input.tsx
├─ lib/                         # framework-agnostic helpers
│  ├─ db.ts
│  └─ utils.ts
├─ hooks/                       # cross-feature client hooks
├─ types/
└─ tsconfig.json                # paths: { "@/*": ["./*"] }

Naming

  • Files: kebab-case.tsx. Components: PascalCase. Hooks: useCamelCase.
  • One default export per route file (page.tsx, layout.tsx). Everything else uses named exports.
  • Co-located helpers go in _components/ or _lib/ (leading underscore — Next.js skips them when routing).

3. Data fetching & mutations

Reads happen on the server

Fetch where the data is rendered. Don't ship a useEffect that calls your own API route just to populate a list — the server already had that data. 

// app/dashboard/page.tsx — Server Component
import { db } from "@/lib/db";

export default async function DashboardPage() {
  const orders = await db.order.findMany({ take: 20 });
  return <OrderTable orders={orders} />;
}

Writes use Server Actions

// app/dashboard/actions.ts
"use server";
import { revalidatePath } from "next/cache";
import { db } from "@/lib/db";

export async function createOrder(formData: FormData) {
  const sku = String(formData.get("sku"));
  await db.order.create({ data: { sku } });
  revalidatePath("/dashboard");
}

// app/dashboard/new-order-form.tsx
"use client";
import { useFormStatus } from "react-dom";
import { createOrder } from "./actions";

function Submit() {
  const { pending } = useFormStatus();
  return <button disabled={pending}>{pending ? "Saving…" : "Save"}</button>;
}

export function NewOrderForm() {
  return (
    <form action={createOrder}>
      <input name="sku" required />
      <Submit />
    </form>
  );
}

Optimistic UI

For mutations where the result is predictable, use useOptimistic + useTransition instead of building your own pending-state machinery.

Don't reach for React Query in App Router projects by default. Server Components + Server Actions cover most CRUD. Add a client cache (TanStack Query, SWR) only when you have genuinely interactive client-driven data — search-as-you-type, polling, optimistic lists too complex for useOptimistic.

4. Component rules

The "use client" directive

It is a boundary, not a label. The first "use client" file in an import chain marks the seam — every module imported from there downward is bundled to the client. Put it as low as possible.

Good 
// page.tsx — server
import { LikeButton } from "./like-button";

export default async function Post({ id }) {
  const post = await getPost(id);
  return (
    <article>
      <h1>{post.title}</h1>
      <p>{post.body}</p>
      <LikeButton postId={id} />
    </article>
  );
}
Bad 
// page.tsx — entire page is client
"use client";
import { useEffect, useState } from "react";

export default function Post({ id }) {
  const [post, setPost] = useState();
  useEffect(() => {
    fetch(`/api/posts/${id}`).then(r => r.json()).then(setPost);
  }, [id]);
  // …
}

Composition over context

Prefer passing Server Components as children into Client Components rather than lifting everything into a client context. This is how you keep server-rendered content inside an interactive shell. 

// providers.tsx — client
"use client";
export function Drawer({ children }: { children: React.ReactNode }) {
  const [open, setOpen] = useState(false);
  return open ? <aside>{children}</aside> : null;
}

// page.tsx — server
<Drawer>
  <ServerOnlyContent />   {/* still rendered on the server */}
</Drawer>

Props

  • Type props explicitly — no implicit any, no React.FC.
  • No prop drilling more than two levels. If you're passing it through, restructure or use composition.
  • Boolean props default to false. Name them affirmatively (disabled, not notEnabled).

repo/
├─ apps/
│  ├─ web/                      # Next.js app
│  └─ admin/                    # second Next.js app
├─ packages/
│  ├─ ui/                       # shared component library
│  ├─ db/                       # Prisma / Drizzle client
│  ├─ config-eslint/
│  ├─ config-tailwind/
│  └─ tsconfig/                 # base tsconfig.json files
├─ pnpm-workspace.yaml
├─ turbo.json
└─ package.json

pnpm-workspace.yaml

packages:
  - "apps/*"
  - "packages/*"

Internal package naming

All internal packages share an @yourorg/ namespace. Reference them from apps with the workspace:* protocol so pnpm always resolves to the local copy. 

// apps/web/package.json
{
  "name": "@yourorg/web",
  "dependencies": {
    "@yourorg/ui": "workspace:*",
    "@yourorg/db": "workspace:*"
  },
  "devDependencies": {
    "@yourorg/config-eslint": "workspace:*",
    "@yourorg/tsconfig": "workspace:*"
  }
}
Internal packages: don't pre-build. For most shared TypeScript packages, export ./src/index.ts directly via the exports field — Next.js compiles them through transpilePackages. Skip the dual tsup build step unless the package is published to npm.

turbo.json baseline

{
  "$schema": "https://turbo.build/schema.json",
  "globalDependencies": [".env"],
  "tasks": {
    "build": {
      "dependsOn": ["^build"],
      "inputs": ["src/**", "package.json", "tsconfig.json"],
      "outputs": [".next/**", "!.next/cache/**", "dist/**"]
    },
    "lint": { "dependsOn": ["^build"] },
    "typecheck": { "dependsOn": ["^build"] },
    "test": { "dependsOn": ["^build"], "outputs": ["coverage/**"] },
    "dev": { "cache": false, "persistent": true }
  }
}

7. Caching & CI

Make tasks cacheable

  • Declare inputs narrowly — don't include README.md or unrelated files; touching them invalidates the cache.
  • Declare outputs precisely — Turbo only restores what you list. For Next.js: ".next/**", "!.next/cache/**".
  • Anything that mutates state (dev, start) must set "cache": false.

Remote cache

Enable Vercel Remote Cache (or a self-hosted one) so CI and every teammate's laptop share build artifacts. A first-time PR build that takes 8 minutes becomes ~30 seconds after the first hit. 

# on a dev machine
pnpm dlx turbo login
pnpm dlx turbo link

CI: affected-only

# .github/workflows/ci.yml (excerpt)
- run: pnpm install --frozen-lockfile
- run: pnpm turbo run lint typecheck test build --filter="...[origin/main]"
  env:
    TURBO_TOKEN: ${{ secrets.TURBO_TOKEN }}
    TURBO_TEAM:  ${{ vars.TURBO_TEAM }}
Filter syntax cheat sheet --filter=@yourorg/web (one package), --filter=@yourorg/web... (package + its deps), --filter=...[origin/main] (only what changed since main).

8. PR checklist

Every PR should pass this before requesting review.

  • pnpm typecheck passes — no any, no @ts-ignore without a comment explaining why.
  • pnpm lint passes with zero warnings (--max-warnings 0).
  • pnpm test passes locally.
  • No orphaned "use client" directives — every client file actually needs hooks, events, or browser APIs.
  • No client-side fetch of data the server could have rendered.
  • No useEffect for derived state (compute it during render) or for data fetching (use a Server Component or Server Action).
  • No new dependency without a one-line justification in the PR description.
  • Server Actions validate input (Zod / Valibot) — never trust FormData directly.
  • Secrets read via process.env.X only inside server code; NEXT_PUBLIC_ only for genuinely public values.
  • Loading and error states exist (loading.tsx / error.tsx) for any new route segment.
  • Accessibility: form inputs have labels, interactive elements are real <button>s, images have alt.
  • Screenshots or a short Loom for any UI change.
